## 1. Identificación de Patrones

### 1.1 Patrones actualmente implementados
| Patrón | Ubicación | Descripción |
|---|---|---|
| Controller Layer | `ProductController.java` | Expone los endpoints HTTP del módulo de inventario |
| Service Layer | `ProductService.java` | Define las operaciones principales del dominio de productos |
| Repository Pattern | `ProductRepository.java` | Encapsula el acceso a datos mediante Spring Data JPA |
| Entity Pattern | `Product.java` | Modela la entidad persistente del dominio |
| DTO Pattern | `ProductCreateRequest`, `ProductUpdateRequest`, `ProductResponse` | Separa entrada, actualización y salida de datos |

### 1.2 Patrones ausentes o por fortalecer
| Patrón faltante | Justificación |
|---|---|
| Implementación formal del servicio con validaciones robustas | Haría más visible la lógica de negocio del dominio |
| Manejo global de excepciones | Mejoraría la consistencia de respuestas ante errores |
| Mapper dedicado | Facilitaría conversión entre entidad y DTO si el sistema crece |
| Validación declarativa | Fortalecería la calidad de los datos de entrada |

## 3. Identificación de Antipatrones

### 3.1 Antipatrón evitado: Fat Controller
**Descripción:** Este antipatrón aparece cuando el controlador concentra demasiadas responsabilidades, como validaciones, lógica de negocio, acceso a base de datos y construcción de respuestas.

**Riesgo en el proyecto:**  
Si `ProductController` manejara directamente consultas, reglas de negocio y persistencia, el código sería más difícil de mantener, probar y extender.

**Relación con este ADR:**  
La existencia de `ProductService` y `ProductRepository` evita que el controlador asuma responsabilidades que no le corresponden.

### 3.2 Antipatrón evitado: Lógica de persistencia en la capa HTTP
**Descripción:** Este antipatrón ocurre cuando la capa que atiende solicitudes web accede directamente a la base de datos o contiene detalles de persistencia.

**Riesgo en el proyecto:**  
Aumentaría el acoplamiento entre la API REST y la infraestructura de datos, haciendo más difícil cambiar o ampliar el sistema.

**Relación con este ADR:**  
La separación en capas permite que el controlador delegue las operaciones al servicio y este, a su vez, al repositorio.

### 3.3 Antipatrón relacionado: Alta dependencia entre componentes
**Descripción:** Ocurre cuando varias partes del sistema dependen de manera rígida unas de otras, sin fronteras claras.

**Riesgo en el proyecto:**  
Un cambio en la forma de persistir, validar o exponer productos podría obligar a modificar múltiples clases simultáneamente.

**Relación con este ADR:**  
La arquitectura en capas reduce este riesgo al distribuir responsabilidades y definir puntos claros de interacción.

### 3.4 Antipatrón detectado: Lógica de negocio poco visible en la implementación
**Descripción:** Aunque existe separación por capas, actualmente solo se observa la interfaz del servicio, sin una implementación detallada visible en el material analizado.

**Riesgo en el proyecto:**  
Esto puede dificultar la evidencia completa de cómo se aplican reglas de negocio, validaciones y manejo de errores.

**Relación con este ADR:**  
El ADR formaliza la estructura en capas actual, pero también deja como mejora fortalecer la implementación concreta del servicio.

## 4. Segmentos de Código con Malas Prácticas

### 4.1 Evidencia del controlador desacoplado
**Archivo:** `ProductController.java`

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

    private final ProductService service;

    public ProductController(ProductService service) {
        this.service = service;
    }

    @PostMapping
    public ProductResponse create(@RequestBody ProductCreateRequest request) {
        return service.create(request);
    }

    @GetMapping("/{id}")
    public ProductResponse getById(@PathVariable Long id) {
        return service.getById(id);
    }

    @GetMapping
    public List<ProductResponse> list(
            @RequestParam(required = false) String material,
            @RequestParam(required = false) String category
    ) {
        return service.list(material, category);
    }

    @PutMapping("/{id}")
    public ProductResponse update(@PathVariable Long id, @RequestBody ProductUpdateRequest request) {
        return service.update(id, request);
    }

    @DeleteMapping("/{id}")
    public void delete(@PathVariable Long id) {
        service.delete(id);
    }
}
```

**Análisis:**  
El controlador expone endpoints y delega la lógica principal al servicio, lo cual es coherente con una arquitectura en capas.

### 4.2 Evidencia de la capa de servicio
**Archivo:** `ProductService.java`

```java
public interface ProductService {
    ProductResponse create(ProductCreateRequest request);
    ProductResponse getById(Long id);
    List<ProductResponse> list(String material, String category);
    ProductResponse update(Long id, ProductUpdateRequest request);
    void delete(Long id);
}
```

**Análisis:**  
La interfaz del servicio actúa como punto intermedio entre el controlador y la persistencia, definiendo claramente las operaciones del dominio.

### 4.3 Evidencia de la capa de persistencia
**Archivo:** `ProductRepository.java`

```java
public interface ProductRepository extends JpaRepository<Product, Long> {
    List<Product> findByMaterialIgnoreCase(String material);
    List<Product> findByCategoryIgnoreCase(String category);
    List<Product> findByMaterialIgnoreCaseAndCategoryIgnoreCase(String material, String category);
}
```

**Análisis:**  
El repositorio encapsula el acceso a datos y evita que la capa HTTP tenga que interactuar directamente con la base de datos.

### 4.4 Mala práctica que se evita
**Mala práctica:** concentrar en una sola clase la recepción de solicitudes, la validación, la lógica del negocio y la persistencia.

**Problema:**
- métodos más extensos y difíciles de leer
- mayor acoplamiento entre presentación y persistencia
- pruebas unitarias más complejas
- menor claridad en responsabilidades

### 4.5 Mejora técnica pendiente
**Mala práctica detectada parcialmente:** ausencia visible de implementación concreta del servicio en el material analizado.

**Problema:**
- dificulta evidenciar reglas de negocio completas
- limita la trazabilidad de decisiones internas del dominio
- reduce la claridad sobre validaciones y manejo de errores

### 4.6 Relación con el ADR
Este ADR formaliza la decisión de estructurar el CRUD de productos por capas.  
La evidencia del código muestra que la separación ya existe en controlador, servicio y repositorio, aunque todavía hay aspectos que pueden fortalecerse en la implementación.

## 5. Diseño e Implementación del ADR

### 5.1 Contexto
El módulo de inventario ya presenta una separación básica por capas mediante `controller`, `service`, `repository`, `entity` y `dto`. Sin embargo, para que esta decisión arquitectónica no quede solo como una organización visual de carpetas, es necesario fortalecer la implementación real de la capa de servicio, centralizando allí la lógica del CRUD y evitando que esa responsabilidad quede dispersa o poco visible.

### 5.2 Decisión
Se decide mantener la arquitectura en capas del módulo de inventario, pero fortalecer su implementación mediante:

- creación o consolidación de la clase `ProductServiceImpl`
- centralización de la lógica del CRUD en la capa de servicio
- delegación limpia desde el controlador hacia el servicio
- uso del repositorio únicamente desde la capa de servicio
- preparación del módulo para incorporar excepciones específicas del dominio

### 5.3 Diseño de la solución

```text
┌──────────────┐
│ Controller   │  ← recibe solicitudes HTTP
└──────┬───────┘
       │
┌──────▼───────────────┐
│ ProductServiceImpl   │  ← lógica del CRUD y del dominio
└──────┬───────────────┘
       │
┌──────▼───────────────┐
│ ProductRepository    │  ← acceso a persistencia
└──────┬───────────────┘
       │
┌──────▼───────┐
│ Product      │  ← entidad persistente
└──────────────┘
```

### 5.4 Implementación propuesta
La implementación de esta decisión consiste en crear o fortalecer la clase `ProductServiceImpl` para que toda la lógica del CRUD quede centralizada en la capa de servicio.

**Interfaz actual del servicio:**

```java
public interface ProductService {
    ProductResponse create(ProductCreateRequest request);
    ProductResponse getById(Long id);
    List<ProductResponse> list(String material, String category);
    ProductResponse update(Long id, ProductUpdateRequest request);
    void delete(Long id);
}
```

**Implementación propuesta del servicio:**

```java
import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class ProductServiceImpl implements ProductService {

    private final ProductRepository repository;

    public ProductServiceImpl(ProductRepository repository) {
        this.repository = repository;
    }

    @Override
    public ProductResponse create(ProductCreateRequest request) {
        Product product = new Product();
        product.setName(request.getName());
        product.setMaterial(request.getMaterial());
        product.setCategory(request.getCategory());
        product.setPrice(request.getPrice());
        product.setStock(request.getStock());
        product.setImageUrl(request.getImageUrl());
        product.setDescription(request.getDescription());

        Product saved = repository.save(product);
        return mapToResponse(saved);
    }

    @Override
    public ProductResponse getById(Long id) {
        Product product = repository.findById(id)
                .orElseThrow(() -> new RuntimeException("Producto no encontrado con id: " + id));
        return mapToResponse(product);
    }

    @Override
    public List<ProductResponse> list(String material, String category) {
        List<Product> products;

        if (material != null && category != null) {
            products = repository.findByMaterialIgnoreCaseAndCategoryIgnoreCase(material, category);
        } else if (material != null) {
            products = repository.findByMaterialIgnoreCase(material);
        } else if (category != null) {
            products = repository.findByCategoryIgnoreCase(category);
        } else {
            products = repository.findAll();
        }

        return products.stream().map(this::mapToResponse).toList();
    }

    @Override
    public ProductResponse update(Long id, ProductUpdateRequest request) {
        Product product = repository.findById(id)
                .orElseThrow(() -> new RuntimeException("Producto no encontrado con id: " + id));

        product.setName(request.getName());
        product.setMaterial(request.getMaterial());
        product.setCategory(request.getCategory());
        product.setPrice(request.getPrice());
        product.setStock(request.getStock());
        product.setImageUrl(request.getImageUrl());
        product.setDescription(request.getDescription());

        Product updated = repository.save(product);
        return mapToResponse(updated);
    }

    @Override
    public void delete(Long id) {
        Product product = repository.findById(id)
                .orElseThrow(() -> new RuntimeException("Producto no encontrado con id: " + id));
        repository.delete(product);
    }

    private ProductResponse mapToResponse(Product product) {
        ProductResponse response = new ProductResponse();
        response.setId(product.getId());
        response.setName(product.getName());
        response.setMaterial(product.getMaterial());
        response.setCategory(product.getCategory());
        response.setPrice(product.getPrice());
        response.setStock(product.getStock());
        response.setImageUrl(product.getImageUrl());
        response.setDescription(product.getDescription());
        return response;
    }
}
```

**Mejora adicional sugerida:**
Como parte de la consolidación de la capa de servicio, puede agregarse una excepción específica del dominio:

```java
public class ProductNotFoundException extends RuntimeException {
    public ProductNotFoundException(String message) {
        super(message);
    }
}
```

Y luego reemplazar en el servicio:

```java
.orElseThrow(() -> new RuntimeException("Producto no encontrado con id: " + id));
```

por:

```java
.orElseThrow(() -> new ProductNotFoundException("Producto no encontrado con id: " + id));
```

Con este cambio:
- la lógica del CRUD queda realmente centralizada en la capa de servicio
- el controlador se mantiene limpio y enfocado en HTTP
- el repositorio solo se usa desde la capa adecuada
- la arquitectura en capas deja de ser solo estructural y pasa a ser funcional

### 5.6 Relación con la arquitectura del sistema
Esta implementación fortalece la arquitectura en capas del módulo de inventario porque convierte la separación estructural en una separación funcional real. La capa de servicio pasa a ser el punto central de la lógica del dominio, el controlador queda desacoplado de la persistencia y el repositorio mantiene su responsabilidad exclusiva de acceso a datos.

## 6. Impacto sobre el Sistema

### 6.1 Beneficios
| Beneficio | Descripción | Impacto |
|---|---|---|
| Orden del código | Cada capa tiene una responsabilidad específica | Alto |
| Mantenibilidad | Facilita cambios sin afectar todas las clases del módulo | Alto |
| Testabilidad | Las capas pueden evaluarse de forma más aislada | Alto |
| Escalabilidad técnica | Facilita agregar validaciones, excepciones y nuevas reglas | Medio |

### 6.2 Riesgos y mitigación
| Riesgo | Probabilidad | Impacto | Mitigación |
|---|---|---|---|
| Mayor cantidad de archivos | Media | Bajo | Mantener una estructura clara y consistente |
| Curva de aprendizaje | Baja | Bajo | Documentar responsabilidades por capa |
| Servicio con implementación incompleta o poco visible | Media | Medio | Fortalecer la clase de implementación del servicio |
| Dependencia de disciplina arquitectónica | Media | Medio | Mantener convenciones en el equipo |

### 6.4 Plan de implementación
**Fase 1: Definición de capas**
- establecer el controlador como punto de entrada HTTP
- definir el servicio como capa intermedia del dominio
- delegar el acceso a datos al repositorio JPA

**Fase 2: Consolidación de responsabilidades**
- mantener el controlador enfocado en recibir solicitudes y responder
- ubicar las reglas del dominio dentro de la implementación del servicio
- aislar consultas y persistencia en el repositorio

**Fase 3: Fortalecimiento del diseño**
- incorporar validaciones de entrada en DTOs y controlador
- implementar manejo centralizado de excepciones
- evaluar uso de mapper dedicado si el módulo crece en complejidad

### 6.5 Consecuencias
**Positivas**
- Mejora la organización interna del módulo  
- Reduce mezcla de responsabilidades  
- Facilita mantenimiento y evolución técnica  
- Da una base más limpia para crecimiento funcional  

**Negativas**
- Incrementa el número de clases del proyecto  
- Requiere disciplina para mantener la separación de capas  
- Puede parecer más complejo en etapas iniciales del desarrollo
