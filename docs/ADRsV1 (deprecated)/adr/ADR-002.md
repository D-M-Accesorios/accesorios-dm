## 3. Identificación de Antipatrones

### 3.1 Antipatrón evitado: Persistencia temporal o improvisada

**Descripción:** Este antipatrón ocurre cuando un módulo funcional usa almacenamiento en memoria, estructuras temporales o soluciones no persistentes para manejar datos que deben conservarse.

**Riesgo en el proyecto:**  
Si el inventario trabajara únicamente con listas temporales o datos en memoria, la información de los productos se perdería al reiniciar la aplicación, afectando la continuidad operativa del sistema.

**Relación con este ADR:**  
La adopción de PostgreSQL permite que la información del inventario permanezca almacenada de forma estable y consistente.

### 3.2 Antipatrón relacionado: Configuración acoplada al código

**Descripción:** Este antipatrón aparece cuando la lógica de conexión a la base de datos se define directamente en clases Java, dificultando el mantenimiento y la flexibilidad del sistema.

**Riesgo en el proyecto:**  
Un cambio de base de datos, credenciales o puerto sería más complejo si la configuración estuviera incrustada en el código fuente.

**Relación con este ADR:**  
La configuración declarada en `application.properties` separa la infraestructura del código de negocio.

### 3.3 Antipatrón relacionado: Base de datos sin modelo de dominio claro

**Descripción:** Se presenta cuando los datos se almacenan sin una estructura alineada con las necesidades reales del negocio.

**Riesgo en el proyecto:**  
Esto dificultaría representar adecuadamente productos, precios, stock y categorías, afectando la integridad del inventario.

**Relación con este ADR:**  
La entidad `Product` y su mapeo a tabla permiten una persistencia estructurada y alineada con el dominio.

### 3.4 Antipatrón detectado: Credenciales visibles en archivo de configuración

**Descripción:** Este antipatrón aparece cuando usuario y contraseña de la base de datos se almacenan directamente en archivos versionados del proyecto.

**Riesgo en el proyecto:**  
Aumenta el riesgo de exposición de credenciales y hace menos segura la configuración del sistema.

**Relación con este ADR:**  
Aunque PostgreSQL fue una decisión correcta, la forma actual de gestionar credenciales debe mejorarse en fases posteriores.

## 4. Segmentos de Código con Malas Prácticas

### 4.1 Evidencia de la decisión de persistencia

**Archivo:** `application.properties`

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/accesoriosdm_inventory
spring.datasource.username=accesoriosdm
spring.datasource.password=accesoriosdm
spring.datasource.driver-class-name=org.postgresql.Driver

spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.open-in-view=false
```

**Análisis:**  
Este fragmento muestra que el microservicio de inventario fue conectado a una base de datos PostgreSQL, usando Spring Boot y JPA para la persistencia del dominio.

### 4.2 Evidencia del modelo persistente

**Archivo:** `Product.java`

```java
@Entity
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 120)
    private String name;

    @Column(nullable = false, length = 40)
    private String material;

    @Column(nullable = false, length = 60)
    private String category;

    @Column(nullable = false, precision = 12, scale = 2)
    private BigDecimal price;

    @Column(nullable = false)
    private Integer stock;

    @Column(length = 300)
    private String imageUrl;

    @Column(length = 500)
    private String description;
}
```

**Análisis:**  
La entidad `Product` representa de forma estructurada la información que debe persistirse en el inventario.

### 4.3 Mala práctica detectada: Credenciales en texto plano

**Mala práctica:** almacenar usuario y contraseña directamente en `application.properties`.

**Problema:**

- expone información sensible
- dificulta despliegues más seguros
- complica manejo de múltiples ambientes
- no sigue una práctica recomendada para producción

### 4.4 Mala práctica detectada: Uso de `ddl-auto=update` como solución permanente

**Mala práctica:** confiar de manera indefinida en actualización automática del esquema desde JPA.

**Problema:**

- puede generar cambios no controlados en la base de datos
- no deja trazabilidad formal de migraciones
- no es la mejor estrategia para ambientes productivos

### 4.5 Relación con el ADR

Este ADR formaliza la decisión de usar PostgreSQL como motor de persistencia del inventario.  
Las malas prácticas señaladas no invalidan la decisión, pero sí muestran aspectos que deben fortalecerse para que la solución sea más robusta.

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El microservicio de inventario ya utiliza PostgreSQL como motor de persistencia, lo que permite almacenar de forma estable la información de los productos. Sin embargo, la configuración actual mantiene las credenciales del datasource escritas directamente en el archivo `application.properties`, lo cual representa una práctica poco segura y poco flexible para futuros ambientes de ejecución.

### 5.2 Decisión

Se decide mantener PostgreSQL como base de datos del módulo de inventario, pero mejorar su implementación mediante la externalización de la configuración sensible usando variables de entorno para la URL, el usuario y la contraseña de conexión.

### 5.3 Diseño de la solución

```text
┌─────────────────────────────────────────────┐
│            INVENTORY-SERVICE                │
├─────────────────────────────────────────────┤
│ Framework: Spring Boot                      │
│ Persistencia: Spring Data JPA               │
│ Motor de BD: PostgreSQL                     │
│ Configuración sensible: Variables de entorno│
│ Base de datos: accesoriosdm_inventory       │
└─────────────────────────────────────────────┘
```

### 5.4 Implementación propuesta

La implementación de esta decisión consiste en modificar la configuración del datasource en `application.properties`, reemplazando los valores fijos por expresiones que permitan leer variables de entorno.

**Configuración actual:**

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/accesoriosdm_inventory
spring.datasource.username=accesoriosdm
spring.datasource.password=accesoriosdm
spring.datasource.driver-class-name=org.postgresql.Driver

spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.open-in-view=false
```

**Configuración propuesta:**

```properties
spring.datasource.url=${DB_URL:jdbc:postgresql://localhost:5432/accesoriosdm_inventory}
spring.datasource.username=${DB_USERNAME:accesoriosdm}
spring.datasource.password=${DB_PASSWORD:accesoriosdm}
spring.datasource.driver-class-name=org.postgresql.Driver

spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.open-in-view=false
```

Con este cambio:

- se conserva PostgreSQL como motor de persistencia
- se eliminan las credenciales fijas del archivo como única fuente de configuración
- se facilita la ejecución del microservicio en distintos ambientes
- se mejora la seguridad y mantenibilidad de la configuración

### 5.5 Estructura relacionada

```text
inventory-service/
├── entity/
│   └── Product.java
├── repository/
│   └── ProductRepository.java
├── resources/
│   └── application.properties
```

### 5.6 Relación con la arquitectura del sistema

## Esta implementación fortalece la decisión arquitectónica de usar PostgreSQL dentro del sistema distribuido, ya que no solo mantiene la persistencia real del inventario, sino que mejora la forma en que el microservicio administra su configuración, preparándolo mejor para ambientes futuros y reduciendo riesgos asociados a credenciales expuestas.

## 6. Impacto sobre el Sistema

### 6.1 Beneficios

| Beneficio                 | Descripción                                                 | Impacto |
| ------------------------- | ----------------------------------------------------------- | ------- |
| Persistencia real         | Los datos sobreviven al reinicio del servicio               | Alto    |
| Integridad de datos       | Los productos y su stock se almacenan de forma estructurada | Alto    |
| Compatibilidad con Spring | PostgreSQL se integra adecuadamente con JPA y Spring Boot   | Alto    |
| Soporte al crecimiento    | Facilita evolución del módulo de inventario                 | Medio   |

### 6.2 Riesgos y mitigación

| Riesgo                              | Probabilidad | Impacto | Mitigación                                        |
| ----------------------------------- | ------------ | ------- | ------------------------------------------------- |
| Credenciales expuestas              | Media        | Alto    | Usar variables de entorno o configuración externa |
| Cambios de esquema no controlados   | Media        | Medio   | Incorporar migraciones versionadas                |
| Dependencia del entorno local       | Media        | Medio   | Documentar instalación y configuración del motor  |
| Sobrecarga de configuración inicial | Baja         | Bajo    | Mantener estructura y documentación claras        |

### 6.4 Plan de implementación

**Fase 1: Configuración de la base de datos**

- definir PostgreSQL como motor del microservicio de inventario
- configurar URL, usuario, contraseña y driver en `application.properties`
- verificar conectividad entre servicio y base de datos

**Fase 2: Integración con el dominio**

- mapear la entidad `Product` a la tabla `products`
- crear el repositorio `ProductRepository` con Spring Data JPA
- validar operaciones CRUD sobre el modelo persistente

**Fase 3: Fortalecimiento de la persistencia**

- migrar credenciales a variables de entorno
- evaluar incorporación de migraciones versionadas
- preparar perfiles de configuración para ambientes futuros

### 6.5 Consecuencias

**Positivas**

- Proporciona persistencia estable para el módulo de inventario
- Mejora la integridad de los datos del catálogo
- Permite pruebas CRUD más confiables
- Da una base sólida para el crecimiento del servicio

**Negativas**

- Requiere configuración adicional del entorno
- Exige mejores prácticas de seguridad para credenciales
- Puede necesitar migraciones más controladas a futuro
