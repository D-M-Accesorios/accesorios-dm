## 1. Identificación de Patrones

### 1.1 Patrones actualmente implementados
| Patrón | Ubicación | Descripción |
|---|---|---|
| DTO Pattern | `ProductCreateRequest.java`, `ProductUpdateRequest.java`, `ProductResponse.java` | Se usan clases separadas para entrada, actualización y salida de datos |
| Separación de contrato API | `ProductController.java` | El controlador no trabaja directamente con la entidad `Product` como request o response |
| Diseño orientado a casos de uso | `dto/` | Cada operación del CRUD usa una estructura de datos específica |
| Desacoplamiento entre capas | `controller/`, `service/`, `entity/`, `dto/` | La comunicación entre capas no expone directamente el modelo persistente |

### 1.2 Patrones ausentes o por fortalecer
| Patrón faltante | Justificación |
|---|---|
| Encapsulamiento de DTOs | Los atributos están definidos como `public`, lo que reduce control sobre los datos |
| Validación declarativa | No se observan anotaciones como `@NotBlank`, `@NotNull`, `@Positive` |
| Mapper dedicado | Facilitaría la conversión entre entidad y DTO si el módulo crece |
| Inmutabilidad o uso de records | Podría mejorar claridad y seguridad del contrato de datos |

## 3. Identificación de Antipatrones

### 3.1 Antipatrón evitado: Exposición directa de entidades
**Descripción:** Este antipatrón ocurre cuando la entidad persistente del dominio se usa directamente como estructura de entrada o salida en la API.

**Riesgo en el proyecto:**  
Si el controlador recibiera o devolviera directamente la entidad `Product`, cualquier cambio en la persistencia afectaría el contrato HTTP del servicio. Esto aumentaría el acoplamiento y reduciría la flexibilidad del sistema.

**Relación con este ADR:**  
El uso de DTOs específicos evita que la API quede atada directamente al modelo persistente.

### 3.2 Antipatrón evitado: Sobrecarga del modelo de dominio
**Descripción:** Aparece cuando una sola clase intenta cumplir varios roles al mismo tiempo, como entidad de base de datos, request de entrada y response de salida.

**Riesgo en el proyecto:**  
La clase del dominio podría crecer con responsabilidades que no le corresponden, dificultando su mantenimiento y claridad.

**Relación con este ADR:**  
Separar `Product` de `ProductCreateRequest`, `ProductUpdateRequest` y `ProductResponse` permite que cada clase tenga un propósito claro.

### 3.3 Antipatrón relacionado: Contrato API poco controlado
**Descripción:** Ocurre cuando no se diferencian los datos necesarios para crear, actualizar y responder, lo que hace más difícil controlar qué información entra y cuál se expone.

**Riesgo en el proyecto:**  
El servicio podría aceptar datos innecesarios o devolver más información de la requerida, afectando seguridad y mantenibilidad.

**Relación con este ADR:**  
La diferenciación de DTOs permite controlar mejor cada operación del CRUD.

### 3.4 Antipatrón detectado: DTOs sin encapsulamiento ni validación
**Descripción:** Aunque los DTOs existen, actualmente sus atributos son públicos y no tienen reglas declarativas de validación.

**Riesgo en el proyecto:**  
Se reduce el control sobre la integridad de los datos y se facilita el ingreso de valores inconsistentes.

**Relación con este ADR:**  
El ADR formaliza la decisión de usar DTOs, pero también deja claro que su implementación puede fortalecerse con encapsulamiento y validaciones.

## 4. Segmentos de Código con Malas Prácticas

### 4.1 Evidencia del DTO de creación
**Archivo:** `ProductCreateRequest.java`

```java
public class ProductCreateRequest {
    public String name;
    public String material;
    public String category;
    public BigDecimal price;
    public Integer stock;
    public String imageUrl;
    public String description;
}
```

**Análisis:**  
Este DTO define los datos que el sistema espera al crear un producto.

### 4.2 Evidencia del DTO de actualización
**Archivo:** `ProductUpdateRequest.java`

```java
public class ProductUpdateRequest {
    public String name;
    public String material;
    public String category;
    public BigDecimal price;
    public Integer stock;
    public String imageUrl;
    public String description;
}
```

**Análisis:**  
Este DTO permite separar la operación de actualización del resto de casos de uso del producto.

### 4.3 Evidencia del DTO de respuesta
**Archivo:** `ProductResponse.java`

```java
public class ProductResponse {
    public Long id;
    public String name;
    public String material;
    public String category;
    public BigDecimal price;
    public Integer stock;
    public String imageUrl;
    public String description;
}
```

**Análisis:**  
Este DTO representa la estructura que el cliente recibe como respuesta desde la API.

### 4.4 Mala práctica detectada: Atributos públicos en los DTOs
**Mala práctica:** definir todos los atributos de los DTOs como `public`.

**Problema:**
- rompe encapsulamiento
- reduce control sobre acceso y modificación de los datos
- dificulta evolucionar el modelo con mayor seguridad
- se aleja de una práctica más limpia de diseño orientado a objetos

### 4.5 Mala práctica detectada: Ausencia de validación declarativa
**Mala práctica:** no usar anotaciones como `@NotBlank`, `@NotNull`, `@Positive` o `@DecimalMin` en los DTOs.

**Problema:**
- permite entrada de datos vacíos o inconsistentes
- desplaza la detección de errores a etapas más tardías
- reduce confiabilidad del contrato API

### 4.6 Relación con el ADR
Este ADR formaliza la decisión de usar DTOs para separar entrada, actualización y salida del dominio de productos.  
Las malas prácticas observadas no invalidan la decisión, pero sí muestran oportunidades claras de mejora en la implementación.

## 5. Diseño e Implementación del ADR

### 5.1 Contexto
El módulo de inventario ya utiliza DTOs para separar la creación, actualización y respuesta de productos. Sin embargo, en la implementación actual los atributos de estos DTOs están definidos como `public` y no cuentan con validaciones declarativas, lo que reduce el control sobre la integridad de los datos que ingresan al sistema.

### 5.2 Decisión
Se decide mantener el uso de DTOs diferenciados para creación, actualización y respuesta de productos, pero fortalecer su implementación mediante:

- encapsulamiento de atributos
- incorporación de validaciones declarativas
- uso de `@Valid` en los endpoints del controlador

### 5.3 Diseño de la solución

```text
┌────────────────────────────┐
│ ProductCreateRequest       │ ← entrada para crear
│ - atributos encapsulados   │
│ - validaciones declarativas│
└────────────────────────────┘

┌────────────────────────────┐
│ ProductUpdateRequest       │ ← entrada para actualizar
│ - atributos encapsulados   │
│ - validaciones declarativas│
└────────────────────────────┘

┌────────────────────────────┐
│ ProductResponse            │ ← salida hacia el cliente
│ - contrato de respuesta    │
└────────────────────────────┘

┌────────────────────────────┐
│ Product                    │ ← entidad persistente interna
└────────────────────────────┘
```

### 5.4 Implementación propuesta
La implementación de esta decisión consiste en modificar los DTOs `ProductCreateRequest` y `ProductUpdateRequest` para que sus atributos dejen de ser `public` y pasen a ser `private`, agregando además validaciones declarativas.

**Implementación actual de `ProductCreateRequest`:**

```java
public class ProductCreateRequest {
    public String name;
    public String material;
    public String category;
    public BigDecimal price;
    public Integer stock;
    public String imageUrl;
    public String description;
}
```

**Implementación propuesta de `ProductCreateRequest`:**

```java
import jakarta.validation.constraints.*;
import java.math.BigDecimal;

public class ProductCreateRequest {

    @NotBlank(message = "El nombre es obligatorio")
    private String name;

    @NotBlank(message = "El material es obligatorio")
    private String material;

    @NotBlank(message = "La categoría es obligatoria")
    private String category;

    @NotNull(message = "El precio es obligatorio")
    @Positive(message = "El precio debe ser mayor que cero")
    private BigDecimal price;

    @NotNull(message = "El stock es obligatorio")
    @Positive(message = "El stock debe ser mayor que cero")
    private Integer stock;

    private String imageUrl;
    private String description;

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public String getMaterial() { return material; }
    public void setMaterial(String material) { this.material = material; }

    public String getCategory() { return category; }
    public void setCategory(String category) { this.category = category; }

    public BigDecimal getPrice() { return price; }
    public void setPrice(BigDecimal price) { this.price = price; }

    public Integer getStock() { return stock; }
    public void setStock(Integer stock) { this.stock = stock; }

    public String getImageUrl() { return imageUrl; }
    public void setImageUrl(String imageUrl) { this.imageUrl = imageUrl; }

    public String getDescription() { return description; }
    public void setDescription(String description) { this.description = description; }
}
```

**Implementación propuesta de `ProductUpdateRequest`:**
Se aplica el mismo principio de encapsulamiento y validación usado en `ProductCreateRequest`.

**Ajuste en el controlador:**

```java
import jakarta.validation.Valid;

@PostMapping
public ProductResponse create(@Valid @RequestBody ProductCreateRequest request) {
    return service.create(request);
}

@PutMapping("/{id}")
public ProductResponse update(@PathVariable Long id, @Valid @RequestBody ProductUpdateRequest request) {
    return service.update(id, request);
}
```

Con este cambio:
- se mantiene la separación entre request y response
- se fortalece el contrato de entrada del CRUD
- se mejora el control sobre los datos que recibe la API
- se reduce el riesgo de datos vacíos o inconsistentes

### 5.5 Estructura relacionada

### 5.6 Relación con la arquitectura del sistema
Esta implementación fortalece la decisión arquitectónica de usar DTOs en el módulo de inventario, ya que mejora el contrato API sin acoplarlo a la entidad persistente. Además, permite que la capa de presentación reciba datos más confiables y mejor estructurados, favoreciendo la mantenibilidad y evolución del sistema distribuido.

## 6. Impacto sobre el Sistema

### 6.1 Beneficios
| Beneficio | Descripción | Impacto |
|---|---|---|
| Desacoplamiento de la API | La entidad persistente no se expone directamente | Alto |
| Mayor control del contrato | Se define qué entra y qué sale en cada operación | Alto |
| Claridad funcional | Crear, actualizar y responder usan estructuras diferenciadas | Alto |
| Evolución más segura | Cambios en la entidad no afectan de inmediato la API | Medio |

### 6.2 Riesgos y mitigación
| Riesgo | Probabilidad | Impacto | Mitigación |
|---|---|---|---|
| Duplicación parcial de campos | Media | Bajo | Mantener una estrategia clara de mapeo |
| DTOs sin validaciones | Alta | Alto | Incorporar anotaciones de validación y `@Valid` |
| DTOs con atributos públicos | Alta | Medio | Usar `private` con getters/setters o Lombok |
| Mayor cantidad de clases | Media | Bajo | Organizar claramente el paquete `dto` |

### 6.4 Plan de implementación
**Fase 1: Consolidación de DTOs**
- mantener un DTO específico para creación
- mantener un DTO específico para actualización
- mantener un DTO específico para respuesta

**Fase 2: Fortalecimiento del contrato**
- encapsular atributos con `private`
- agregar getters/setters o usar Lombok
- definir con claridad qué campos deben recibirse y cuáles deben responderse

**Fase 3: Mejora de calidad de datos**
- incorporar anotaciones de validación en los DTOs
- usar `@Valid` en el controlador
- evaluar la incorporación de un mapper dedicado para convertir entre DTO y entidad

### 6.5 Consecuencias
**Positivas**
- Mejora la claridad del contrato API  
- Reduce el acoplamiento con la entidad persistente  
- Facilita separar responsabilidades entre entrada, actualización y salida  
- Prepara mejor el módulo para validaciones y evolución futura  

**Negativas**
- Aumenta la cantidad de clases del proyecto  
- Requiere mapeo entre DTOs y entidad  
- Puede generar duplicación parcial de campos si no se gestiona bien

## 8. Decisión Final
Se aprueba el uso de DTOs diferenciados para creación, actualización y respuesta de productos en el módulo de inventario de Accesorios D&M, porque esta decisión mejora el control del contrato API, reduce el acoplamiento con la entidad persistente y facilita la evolución del sistema.
