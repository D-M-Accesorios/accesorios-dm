## 2. Análisis del Proyecto y ADRs Potenciales

### 2.1 Estado actual de la arquitectura

El módulo de inventario no solo requiere operaciones CRUD básicas, sino también consultas funcionales para el catálogo.  
Por ello se diseñó un contrato REST capaz de crear, consultar, actualizar, eliminar y listar productos con filtros opcionales por material y categoría.

```text
┌──────────────────────────────────────────────────────────────┐
│                CONTRATO ACTUAL DEL INVENTARIO               │
├──────────────────────────────────────────────────────────────┤
│ POST   /api/products                                         │
│ GET    /api/products/{id}                                    │
│ GET    /api/products                                         │
│ PUT    /api/products/{id}                                    │
│ DELETE /api/products/{id}                                    │
│                                                              │
│ Filtros opcionales:                                          │
│ - material                                                   │
│ - category                                                   │
│                                                              │
│ Consultas soportadas por repositorio:                        │
│ - findByMaterialIgnoreCase                                   │
│ - findByCategoryIgnoreCase                                   │
│ - findByMaterialIgnoreCaseAndCategoryIgnoreCase              │
└──────────────────────────────────────────────────────────────┘
```

## 3. Identificación de Antipatrones

### 3.1 Antipatrón evitado: CRUD rígido sin capacidad de consulta útil

**Descripción:** Este antipatrón aparece cuando una API solo permite operaciones básicas de mantenimiento, pero no responde a necesidades reales del negocio, como filtrar productos para el catálogo.

**Riesgo en el proyecto:**  
Si el inventario solo permitiera crear, leer por id, actualizar y eliminar, la integración con una web o una app móvil sería limitada, ya que no existirían consultas útiles para mostrar productos por atributos comerciales.

**Relación con este ADR:**  
Los filtros opcionales por material y categoría hacen que el contrato REST sea más funcional para el dominio de accesorios.

### 3.2 Antipatrón evitado: Filtrado manual en memoria

**Descripción:** Este antipatrón ocurre cuando el sistema obtiene todos los registros y luego los filtra en la lógica de aplicación, en lugar de delegar el filtrado al repositorio o a la base de datos.

**Riesgo en el proyecto:**  
A medida que el inventario crezca, cargar todos los productos para filtrarlos manualmente sería menos eficiente y menos mantenible.

**Relación con este ADR:**  
Las consultas derivadas del repositorio permiten filtrar desde la capa de persistencia.

### 3.3 Antipatrón relacionado: Contrato API ambiguo

**Descripción:** Se presenta cuando no está claramente definido qué endpoints existen, qué parámetros aceptan y cómo responden.

**Riesgo en el proyecto:**  
Dificulta el consumo de la API desde otros módulos y puede generar inconsistencias entre frontend, servicio y pruebas.

**Relación con este ADR:**  
El diseño explícito de endpoints CRUD y filtros mejora la claridad del contrato.

### 3.4 Antipatrón detectado: Parámetros de filtro sin tipado fuerte

**Descripción:** Actualmente `material` y `category` se manejan como `String`, lo que permite errores de escritura o valores no válidos.

**Riesgo en el proyecto:**  
Se pueden enviar consultas con valores inconsistentes, afectando confiabilidad y claridad de la API.

**Relación con este ADR:**  
Aunque el contrato REST ya existe y es útil, una mejora futura sería fortalecer estos parámetros con enums y validaciones.

## 4. Segmentos de Código con Malas Prácticas

### 4.1 Evidencia del contrato REST

**Archivo:** `ProductController.java`

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

    private final ProductService service;

    public ProductController(ProductService service) {
        this.service = service;
    }

    @PostMapping
    public ProductResponse create(@RequestBody ProductCreateRequest request) {
        return service.create(request);
    }

    @GetMapping("/{id}")
    public ProductResponse getById(@PathVariable Long id) {
        return service.getById(id);
    }

    @GetMapping
    public List<ProductResponse> list(
            @RequestParam(required = false) String material,
            @RequestParam(required = false) String category
    ) {
        return service.list(material, category);
    }

    @PutMapping("/{id}")
    public ProductResponse update(@PathVariable Long id, @RequestBody ProductUpdateRequest request) {
        return service.update(id, request);
    }

    @DeleteMapping("/{id}")
    public void delete(@PathVariable Long id) {
        service.delete(id);
    }
}
```

**Análisis:**  
El controlador define de forma clara las operaciones CRUD y el endpoint de listado con filtros opcionales, lo cual responde mejor a necesidades reales del inventario.

### 4.2 Evidencia de los filtros soportados por repositorio

**Archivo:** `ProductRepository.java`

```java
public interface ProductRepository extends JpaRepository<Product, Long> {
    List<Product> findByMaterialIgnoreCase(String material);
    List<Product> findByCategoryIgnoreCase(String category);
    List<Product> findByMaterialIgnoreCaseAndCategoryIgnoreCase(String material, String category);
}
```

**Análisis:**  
El repositorio implementa consultas específicas por material y categoría, evitando filtrado manual en memoria.

### 4.3 Mala práctica detectada: Parámetros como `String`

**Mala práctica:** manejar `material` y `category` como texto libre.

**Problema:**

- permite valores mal escritos
- no existe tipado fuerte
- puede generar inconsistencias funcionales
- dificulta validación del dominio

### 4.4 Mala práctica detectada: Respuesta HTTP poco explícita en borrado

**Mala práctica:** usar `void` en el endpoint `DELETE` sin un contrato más explícito del código de respuesta.

**Problema:**

- reduce claridad del contrato REST
- hace menos evidente el uso esperado de `204 No Content`
- puede generar dudas en consumidores de la API

### 4.5 Mala práctica detectada: Falta de validación visible en entradas

**Mala práctica:** no se observa uso de `@Valid` ni validación declarativa en los endpoints del controlador.

**Problema:**

- permite entrada de datos inconsistentes
- reduce robustez del contrato API
- desplaza los errores a capas posteriores

### 4.6 Relación con el ADR

Este ADR formaliza la decisión de diseñar el módulo de inventario con una API REST CRUD útil para el negocio, incluyendo filtros opcionales.  
Las malas prácticas detectadas no invalidan la decisión, pero sí muestran oportunidades de mejora para fortalecer el contrato.

## 5. Diseño e Implementación del ADR

### 5.1 Contexto

El módulo de inventario ya expone un contrato REST con operaciones CRUD y un endpoint de listado con filtros opcionales por material y categoría. Sin embargo, la implementación actual puede fortalecerse para que el contrato HTTP sea más claro, especialmente en el manejo de respuestas y códigos de estado.

### 5.2 Decisión

Se decide mantener el contrato REST del módulo de inventario con operaciones CRUD y filtros opcionales, pero fortalecer su implementación mediante:

- uso de `ResponseEntity` en los endpoints
- definición más explícita de códigos de estado HTTP
- mejora de la claridad del contrato para consumidores de la API

### 5.3 Diseño de la solución

```text
POST   /api/products                       → 201 Created
GET    /api/products/{id}                  → 200 OK
GET    /api/products                       → 200 OK
GET    /api/products?material=RODIO        → 200 OK
GET    /api/products?category=COLLARES     → 200 OK
GET    /api/products?material=RODIO&category=COLLARES → 200 OK
PUT    /api/products/{id}                  → 200 OK
DELETE /api/products/{id}                  → 204 No Content
```

### 5.4 Implementación propuesta

La implementación de esta decisión consiste en ajustar `ProductController` para que los endpoints devuelvan respuestas HTTP más explícitas usando `ResponseEntity`.

**Implementación actual del controlador:**

```java
@PostMapping
public ProductResponse create(@RequestBody ProductCreateRequest request) {
    return service.create(request);
}

@GetMapping("/{id}")
public ProductResponse getById(@PathVariable Long id) {
    return service.getById(id);
}

@GetMapping
public List<ProductResponse> list(
        @RequestParam(required = false) String material,
        @RequestParam(required = false) String category
) {
    return service.list(material, category);
}

@PutMapping("/{id}")
public ProductResponse update(@PathVariable Long id, @RequestBody ProductUpdateRequest request) {
    return service.update(id, request);
}

@DeleteMapping("/{id}")
public void delete(@PathVariable Long id) {
    service.delete(id);
}
```

**Implementación propuesta del controlador:**

```java
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;

@PostMapping
public ResponseEntity<ProductResponse> create(@RequestBody ProductCreateRequest request) {
    ProductResponse response = service.create(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(response);
}

@GetMapping("/{id}")
public ResponseEntity<ProductResponse> getById(@PathVariable Long id) {
    return ResponseEntity.ok(service.getById(id));
}

@GetMapping
public ResponseEntity<List<ProductResponse>> list(
        @RequestParam(required = false) String material,
        @RequestParam(required = false) String category
) {
    return ResponseEntity.ok(service.list(material, category));
}

@PutMapping("/{id}")
public ResponseEntity<ProductResponse> update(@PathVariable Long id, @RequestBody ProductUpdateRequest request) {
    return ResponseEntity.ok(service.update(id, request));
}

@DeleteMapping("/{id}")
public ResponseEntity<Void> delete(@PathVariable Long id) {
    service.delete(id);
    return ResponseEntity.noContent().build();
}
```

Con este cambio:

- el endpoint de creación responde con `201 Created`
- los endpoints de consulta y actualización responden con `200 OK`
- el endpoint de eliminación responde con `204 No Content`
- el contrato de la API se vuelve más claro para frontend, Postman y futuras integraciones

### 6.4 Plan de implementación

**Fase 1: Definición del contrato REST**

- establecer los endpoints CRUD del módulo de inventario
- definir el endpoint de listado con filtros opcionales
- mantener consistencia en rutas y nombres de parámetros

**Fase 2: Soporte desde la capa de servicio y repositorio**

- delegar la operación de listado al servicio
- implementar consultas derivadas en el repositorio
- mantener la lógica de filtrado alineada con el dominio del producto

**Fase 3: Fortalecimiento del contrato**

- incorporar validación de parámetros
- considerar enums para material y categoría
- evaluar el uso de `ResponseEntity` y documentación formal de endpoints

### 6.5 Consecuencias

**Positivas**

- Mejora la utilidad funcional del microservicio
- Facilita búsquedas reales para el catálogo
- Reduce la necesidad de filtrado manual fuera del servicio
- Prepara mejor la API para consumo por otras interfaces

**Negativas**

- Requiere mayor cuidado en validación de parámetros
- Puede crecer en complejidad si se agregan más filtros
- Necesita fortalecerse con tipado y documentación más formal
